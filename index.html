# Sentinel 後台專案

## Tech Stack
- Framework: Nuxt 3 + Monorepo (pnpm workspace)
- UI: Vuetify 3 / TypeScript
- 狀態管理: Pinia
- API: Laravel 12，RESTful

## 命名慣例
- 元件用 PascalCase（UserTable.vue）
- Composable 用 use 前綴（useUserList.ts）
- Store 用 camelCase + Store 後綴（userStore）

## 注意事項
- 不直接在元件裡呼叫 API，一律透過 composable
- 不使用 any 型別，必須明確定義
- 修改 packages/shared 前先確認組長

## 時間處理
- 統一使用 useDateTime composable
- 送後端參數格式：ISO 8601
效果：不用每次開 session 都解釋「我們用 Nuxt 3 + Pinia，命名要這樣...」，AI 一開始就知道了。
路徑：.claude/skills/pr-review/SKILL.md
---
name: pr-review
description: 進行 Vue 元件 Code Review，確認型別、架構、命名
---

進行元件審查時，請依序確認：

1. Props 是否有明確的 TypeScript 型別定義
2. Emit 事件命名是否 kebab-case
3. API 呼叫是否透過 composable
4. 是否有不必要的 any 型別
5. JSDoc 說明是否完整

回傳格式：
問題清單（嚴重程度排序）+ 建議改法

輸入 /pr-review 就觸發，不用每次重新說明要查什麼。

settings.json → 存 .vue 自動執行 ESLint
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write(*.vue)",
      "hooks": [{
        "type": "command",
        "command": "npx eslint --fix $file"
      }]
    }],
    "Notification": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "osascript -e 'display notification \"Claude 完成了\" with title \"Claude Code\"'"
      }]
    }]
  }
}

存完 .vue 自動跑 ESLint 修正，任務跑完自動桌面通知，不用一直盯著 terminal。

路徑：.claude/agents/codebase-explorer.md

Subagent 是一個在獨立 context window 裡執行的 AI 分身，完成任務後只把摘要回傳給你的主對話。它最大的用途是處理「需要大量讀取程式庫、查找檔案、探索架構」這類工作——這些動作如果放在主對話裡跑，會把 context 塞爆，影響後續開發的精準度。 

---
name: codebase-explorer
description: 掃描並分析專案程式庫結構，適合在需要理解新功能影響範圍、
             或接手陌生模組時使用。不會修改任何檔案。
tools: Read, Glob, Grep
---

你是一個唯讀的程式庫分析助理。

任務收到後，依序執行：
1. 掃描相關目錄，找出與任務相關的檔案
2. 理解各模組的職責與相互依賴關係
3. 找出潛在的影響範圍（哪些地方引用了目標模組）
4. 整理成簡潔的摘要回傳給主對話

輸出格式：
- 相關檔案清單（含路徑）
- 模組依賴關係說明
- 潛在影響範圍
- 建議的修改切入點

# 此 agent 只使用 Read / Glob / Grep，不會執行任何寫入操作
使用時機：當你要修改一個陌生的模組、或評估一個新功能的影響範圍時，直接對主對話說「用 codebase-explorer 幫我分析 UserStore 的依賴」，Subagent 在背景跑完後只回傳一份摘要，主對話的 context 完全不受影響，後續開發仍然乾淨。
與 Skill 的差異：Skill 是在主對話的 context 裡執行，Subagent 是完全隔離的獨立執行空間。需要大量探索、讀取很多檔案時用 Subagent；需要直接和主對話互動、邊分析邊修改時用 Skill。
路徑：<project>/.claude/rules/api-rules.md

Rules 是 CLAUDE.md 的進階用法，讓你針對特定路徑的檔案套用不同的規範，而不是讓所有規則對所有檔案都生效。YAML frontmatter 的 paths 欄位定義觸發條件，只有在 Claude 處理符合路徑的檔案時，這條規則才會被載入。 

---
paths:
  - "src/api/**/*.ts"
  - "server/routes/**/*.ts"
---

# API 端點開發規範

撰寫 API 相關程式碼時，務必遵守以下規範：

## 命名
- 路由路徑使用 kebab-case：/user-profile 而非 /userProfile
- Controller 方法使用動詞開頭：getUser、createOrder、deleteSession

## 錯誤處理
- 所有 API 回應統一使用 ApiResponse<T> 型別
- 錯誤碼對照表在 src/constants/errors.ts，不可自行新增錯誤碼
- 必須捕捉並記錄例外，不允許 silent catch

## 輸入驗證
- 所有 request body 必須經過 Zod schema 驗證
- 驗證 schema 放在對應 route 檔案的旁邊（同目錄）

## 文件
- 每個端點必須附上 OpenAPI 格式的 JSDoc 說明
為什麼不直接放進 CLAUDE.md？如果把 API 規範放進主 CLAUDE.md，Claude 在寫前端元件時也會把這些規則載入 context，既浪費 token 又可能產生不必要的干擾。Rules 讓你的規範精準地只在對的地方生效——只有當 Claude 正在處理 src/api/**/*.ts 這類檔案時，這條規則才會出現。
存放位置：放在 <project>/.claude/rules/ 目錄下，支援 symlink，可以讓多個專案共用同一份規則檔。沒有路徑 frontmatter 的規則檔則在所有情況下都會載入，效果等同於 CLAUDE.md。

沒有配置的狀態
每次 session 都要從零開始建立情境，AI 對你的專案架構一無所知。

工作流依賴個人記憶和習慣，重複性操作（格式化、規範確認）容易被遺忘或跳過。

長任務執行期間注意力被佔據，context window 在大量探索後快速消耗，影響後續開發效率。

整體感受：每天大約 30–60 分鐘消耗在「讓 AI 理解我是誰、我在做什麼」這件事上。
配置完成後的狀態
CLAUDE.md 讓 AI 每次啟動就具備完整的專案知識，直接進入開發節奏，不需要反覆解釋。

Hooks 把格式化、通知等瑣碎動作自動化，從「需要記得做」變成「系統保證會做」。

Skills 和 Subagents 讓不同類型的任務有對應的專屬處理方式，主對話保持乾淨聚焦。

整體感受：AI 從「你在教的工具」變成「了解你工作方式的協作夥伴」，節省的不只是時間，而是認知負擔。
💡
配置的核心價值不是「節省幾分鐘」
好的 AI 配置讓你的工作流從「每次對話都是全新開始」變成「AI 持續理解你的專案脈絡」。這份理解會隨時間累積，讓 AI 在你的開發環境裡越用越順。這正是這次競賽想看到的——不是炫技，而是真正讓你工作更省力的那套設定。
